## P1：将客户端重构为可配置的多设备分阶段数据管线

- 状态：待设计、待实施
- 记录日期：2026-07-15
- 当前入口：`cpp/client/src/LidarClientWorker.cpp`

### 背景与问题

当前 `LidarClientWorker` 在同一个 `QThread` 中完成 `QTcpSocket` 收包、JSONL
拆帧、协议解析、设备状态合并、扫描周期监控、L0-L2 科学算法和显示快照构建。
`FrameProcessor::handle_frame()` 一旦耗时增长，该线程就不能及时返回 Qt 事件循环继续处理
`readyRead`、断线、命令响应等事件。接入激光雷达、地面 PM 仪和温度传感器等多个设备后，
任一设备或算法变慢还会影响其他设备。

### 目标

将客户端拆成可配置、可观测且有明确所有权的四层管线：

```text
DeviceManager（设备会话、启停和健康状态）
  -> DeviceSession + ITransport（TCP / 串口 / 后续厂商 SDK，I/O 执行上下文）
  -> IFrameDecoder / IProtocolCodec（JSON / 字节流 / 厂商协议，保持设备内顺序）
  -> 有界消息队列（背压、优先级、丢弃策略和统计）
  -> ProcessorScheduler（雷达算法等独立工作线程或线程池）
  -> 不可变结果快照 -> GUI 线程
```

配置至少描述：`device_id`、设备类型、传输类型及参数、协议类型及参数、处理管线、
队列容量、超时/重连策略和过载策略。TCP 与串口的差异由 `ITransport` 实现封装，
协议选择由 codec/decoder 注册表或工厂完成，业务处理器不依赖具体通信方式。

### 线程模型约束

- 每个设备拥有独立 `DeviceSession`。第一版设备数量较少时，可以让每个通信设备拥有
  独立 I/O 工作线程，以获得清晰的线程亲和性和故障隔离；接口设计不得强制未来所有
  非阻塞 socket 都永久采用“一设备一线程”。
- 不默认采用“一种协议一个固定线程”。协议是解析策略，不是天然的线程边界。轻量、
  有状态的流式拆帧可在对应 I/O 上下文顺序执行；确实耗 CPU 或调用阻塞厂商库时，
  再投递到有界的解析执行器，并保证同一设备内的消息顺序。
- 雷达科学算法必须离开通信事件循环。`FrameProcessor` 是有周期状态的顺序处理器，
  初期由一个专用算法线程独占；确认任务可并行和状态可安全分片后，才能改为线程池。
- 跨线程只传递带有 `device_id`、设备时间、主机接收时间、序号和消息类型的不可变消息，
  大块原始数组使用移动语义或只读共享所有权，避免重复复制和数据竞争。
- I/O 线程不得等待解析或算法完成；通信/解析与算法之间使用“有容量上限的线程安全
  处理队列”，所有跨阶段队列禁止无界堆积。不要仅依赖 Qt 排队信号承载高流量数据，
  因为其事件队列没有本管线可管理的容量和过载策略。

### 算法慢于数据到达时的处理策略

线程拆分后必须显式处理这种情况：算法线程仍在处理扫描周期 A，而通信和解析阶段已经
得到周期 B、C。队列只能吸收短时突发，不能解决长期吞吐不足。如果算法的平均处理耗时
持续大于扫描周期，任何有限队列最终都会满，届时必须降低采集频率、优化/并行算法、让
设备暂停、把原始数据落盘后异步处理，或者接受丢弃数据。

- 雷达处理队列优先以完整 `ScanCycleBatch` 为元素，同时限制周期数量和累计字节数。
  不能在队列满时随机丢一条射线，否则后续算法可能把残缺周期当成有效输入。接收侧需要
  根据周期 ID、时间戳或 `heartbeat` 维护轻量周期边界，并把缺帧/丢周期原因写入 QC。
- 提供 `realtime_latest` 模式：默认仅排队 2-3 个待处理完整周期；队列满时丢弃最旧的
  未处理周期、保留最新周期，适合实时 GUI。必须累计 `dropped_cycle_count`，并向界面和
  日志发布数据缺口，不能静默丢弃。
- 提供 `lossless_recording` 模式：完整原始协议帧先顺序写入可追溯文件，再投递在线算法。
  在线算法队列过载时仍可丢弃旧周期，但能够从原始记录离线重放补算。需要同时处理磁盘
  写入失败、磁盘空间不足和记录队列过载，不能把“已接收”等同于“已可靠落盘”。
- 设备协议支持时可提供 `controlled_acquisition` 模式：处理队列达到高水位后发送
  `pause`，降至低水位后发送 `resume`。TCP 窗口或串口 RTS/CTS 只能作为传输层辅助，
  不能假定所有设备都支持可靠业务流控。
- 状态、告警和命令响应使用独立的高优先级通道，不能排在大体积雷达数据之后，也不能
  套用雷达周期的丢弃策略。
- 若测量表明单射线算法可以并行，再把 `FrameProcessor` 拆成无状态/独立的射线计算任务
  和顺序的周期聚合器。当前有状态 `FrameProcessor` 不允许被多个算法线程并发调用。

### 必须先明确的策略

- 背压：雷达原始帧、状态/告警、命令响应不能共用同一种丢弃规则。控制与告警消息优先，
  数据消息根据产品要求选择“阻止采集”“丢最旧”“保留最新”或“落盘后处理”。
- 时序：统一设备时钟与主机接收时钟，处理乱序、重复、断线重连后的序号重置，并定义
  PM、温度和雷达扫描周期的时间对齐窗口。
- 生命周期：定义配置加载、会话启动、重连、停机排空、强制取消和热更新行为；Qt
  `QObject` 必须在其所属线程创建和销毁，不能留下跨线程悬空回调。
- 命令链路：接收管线之外，还要支持按 `device_id` 路由命令、关联请求与响应、超时和重试，
  避免 `DeviceManager` 只管理输入数据。
- 可观测性：按设备、阶段记录连接状态、接收速率、解析错误、队列深度、丢帧数、处理
  延迟和端到端延迟，能够判断瓶颈是在设备、协议还是算法。
- 故障隔离：单设备协议异常、队列过载或算法异常不能导致其他设备会话退出；协议解析必须
  限制帧长、字段数量和缓存上限。

### 管理类职责边界

- `DeviceManager` 只负责配置、会话注册、启停、路由和健康汇总，不直接收包或执行算法。
- `TransportFactory`/注册表根据配置创建 `TcpTransport`、`SerialTransport` 等实现。
- `ProtocolCodecFactory`/注册表根据配置为每个会话创建独立 decoder 实例；有状态解析器
  不得被多个设备无保护共享。
- `ProcessorScheduler` 管理处理器执行上下文和队列；具体雷达、PM、温度处理逻辑仍放在
  各自处理器中，避免形成新的“万能管理类”。

### 分阶段实施

1. 为当前单 TCP 雷达链路补充接收、解析、队列和算法耗时基线，增加慢算法与突发数据测试。
2. 在不改变现有 JSONL 行为的前提下抽出 `ITransport`、流式 framer/codec 和统一消息信封。
3. 在通信与 `FrameProcessor` 之间加入有界队列，把雷达处理及显示快照构建迁到独立线程，
   实现 `realtime_latest` 过载策略，验证算法变慢时 socket、状态帧和命令响应仍可及时处理。
4. 引入 `DeviceSession`/`DeviceManager` 和配置校验，接入串口 PM、温度设备及其 decoder。
5. 增加 `lossless_recording` 和设备支持时的 `controlled_acquisition`，再补充多设备同时
   运行、重连、停机、队列过载、乱序/坏帧和时间对齐的集成测试。

### 验收条件

- 仅修改配置即可同时接入一个 TCP 激光雷达、一个串口 PM 仪和一个串口温度传感器。
- 新增传输或协议实现时，不修改现有设备处理器和 GUI 的核心逻辑。
- 人为让雷达算法持续慢于数据到达速度时，其他设备仍能收包，控制/告警链路仍可响应，
  内存保持在配置上限内，并能看到明确的排队和丢弃指标。
- `realtime_latest` 过载测试只丢弃完整待处理周期，保留最新周期并产生可观测的数据缺口；
  `lossless_recording` 过载测试能够从原始记录重放被在线处理跳过的周期。
- 同一设备的协议消息保持顺序；跨设备融合使用显式时间窗口，不依赖线程调度先后。
- 客户端可以在任意连接阶段可靠停止，所有线程、socket、串口、队列和处理器均被回收。
- 单元测试可通过内存/仿真 transport 注入数据，不依赖真实硬件或固定线程时序。
