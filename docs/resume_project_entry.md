# 简历项目版本

## 项目名称

大气激光雷达颗粒物反演与站点级监控系统

## 一句话版本

围绕大气激光雷达颗粒物监测场景，独立搭建从 L0-L2 处理、Fernald 反演、湿度修正、PM 标定、热点检测到公开 Cloudnet 数据接入的端到端算法系统，并实现面向 YLJ5 公开规格的 C++ 实时仿真设备、四通道协议和控制闭环。

## 简历长版本

### 项目简介

面向扬尘和区域污染监测场景，设计并实现了一个可评测、可演示、可继续工程化的大气激光雷达颗粒物反演系统。项目第一阶段用可控仿真数据打通回波模拟、反演、标定和热点检测主链；第二阶段接入 Cloudnet 公开激光雷达廓线和 Open-Meteo 地面 PM / 气象数据，构建“真实 stare + 合成 PPI”混合数据链，并补齐站点级偏移校准、漂移监控和失败案例回放，使系统从“能跑 demo”提升到“具备真实数据接入能力、可解释、可验证”的算法原型。

### 职责与贡献

1. 设计并实现大气 LiDAR 最小可运行算法主链，完成回波预处理、Fernald 近似反演、湿度修正、PM2.5 / PM10 标定、热点检测和 ENU 空间定位。
2. 接入 Cloudnet Bucharest 公开样例与 Open-Meteo 地面观测，完成 2025-04-23 单日 18 个真实 stare 时间步、234 条 profile 的混合数据处理闭环，形成可复现的真实公开数据接入方案。
3. 贯通 site_id、多源时间对齐和特征表构建，在 PM 标定阶段增加站点级偏移校准，使统一模型可以保留站点差异并输出可解释校准参数。
4. 建立残差窗口漂移监控和失败案例套件，对背景光抬升、overlap 失配、湿度突增等场景做回放评估，把算法结果从“离线能跑”推进到“上线前可监控、可解释”。
5. 完成配置化实验、批处理、轻量 API、静态 dashboard、单元测试、敏感性分析和消融分析，形成算法、数据、评测、演示一体化闭环。
6. 排查并修复 Windows 非 ASCII 路径下 netCDF4 无法打开 Cloudnet 文件的问题，增加 ASCII 镜像缓存路径，保证公开样例在当前环境下稳定可跑。
7. 基于官网和型号匹配的采购材料建立 YLJ5 规格模型，实现 532 nm、3.75 m、20 km、双望远镜×双偏振四接收通道、惰性周期生成、TCP JSONL 协议和启停/暂停/扫描配置状态机，并明确区分公开边界、仿真假设和未知厂商协议。

### 技术栈

- C++20、CMake、TCP JSONL、线程同步、惰性数据生成
- Python 3.11、netCDF4、JSON 配置、轻量 HTTP API、批处理脚本、单元测试
- 自研 LiDAR 预处理、Fernald 反演、湿度修正、PM 标定、热点检测、ENU 映射模块
- Cloudnet 公开激光雷达数据、Open-Meteo 地面 PM / 气象数据、多源时序对齐

### 结果量化

模拟基线结果：

- PM2.5：RMSE 2.555，$R^2$ 0.819
- PM10：RMSE 2.863，$R^2$ 0.892
- 热点检测：Precision 0.628，Recall 0.754，F1 0.685
- 平均处理时延 0.122 ms / profile，吞吐量 4721 profiles/s

公开真实数据接入结果：

- Cloudnet Bucharest 样例完成 18 个真实 stare profile 与 216 个合成 PPI profile 的混合处理，总计 234 条 profile
- 生成站点级校准参数、漂移监控摘要和失败案例报告，当前 public sample 漂移告警数为 0
- 在真实公开样例链路下平均处理吞吐量 3753 profiles/s

说明：公开真实样例阶段的重点是验证真实 LiDAR 接入、监控和工程闭环，不把单日 hybrid 样例指标包装成最终业务精度。

YLJ5 实时设备工程验证：

- 默认公开规格周期包含 181 条原始射线，每条 5334 个距离门和四条物理接收通道；
- 扫描任务按 0 度水平/5 度锥形逐周期轮换，并在每周期加入 90 度垂直观测；
- 开发机双周期全规格正演 362 条射线约 13.74 秒，最大常驻内存约 155.4 MiB；
- C++ 测试覆盖规格校验、四通道 JSONL 往返、暂停/恢复、扫描配置拒绝和 TCP 命令结果；
- 上述数字是仿真软件工程基线，不是 YLJ5 实机性能或计量精度。

## 面试展开版本

如果面试官追问，可以按下面顺序展开：

1. 为什么真实数据阶段先采用“真实 stare + 合成 PPI”的 hybrid 策略，而不是一开始就做完整真实扫描真值闭环。
2. 为什么 PM 标定除了全局模型，还需要站点级偏移校准，以及这种设计在多站部署里如何扩展。
3. 漂移监控为什么要看残差窗口偏移，而不是只看单点误差或一次性离线指标。
4. 为什么热点检测不能只用绝对 PM 阈值，而要叠加扫描内相对异常和连通域约束。
5. 在真实 Cloudnet 接入过程中，如何定位并修复 Windows 非 ASCII 路径下 netCDF4 读取失败的问题。
6. 如果把当前原型继续做成线上系统，还需要补哪些能力，比如多站点在线校准、数据回放、数据库、流式处理和 CI。
