# YLJ5 厂商协议还原边界

## 结论

公开网页和采购文件足以建立设备能力与数值边界，不能推出厂商私有实时帧。当前项目的
TCP JSONL 是可测试的数字孪生协议，`vendor_wire_protocol_known=false`，不得写成厂家
协议已还原。

## 当前已有证据

- 官网：YLJ5 型号、双望远镜近远场、同步相机、碳纤维外壳、IP66 和应用方式；
- 采购文件：`AGHJ-I-LIDAR(MPL)`、532 nm、3.75 m、至少 20 km、偏振、云台范围、
  扫描数量、积分脉冲和水平/垂直/锥形模式；
- 相近公开论文：固定低仰角、全方位、2 度步进等扫描方法。

这些证据支持规格校验和仿真行为，不支持推断帧头、字节序、校验、命令字或产品公式。
详细来源见 [YLJ5 保真说明](ylj5_fidelity.md)。

## 当前项目帧

| 帧 | 作用 |
|---|---|
| `status` | 身份、公开能力、当前扫描计划、证据和未知项 |
| `telemetry` | 明确标为合成的设备运行量 |
| `lidar_raw` | 采集时序、距离轴、拼接主通道和四物理通道 |
| `camera` | 同步相机能力与指向，不含伪造图像 |
| `lidar_product` | 未标定快速摘要，不作为客户端 PM 输入 |
| `ground_obs` | 可选合成地面观测 |
| `heartbeat` | 扫描周期完成边界 |
| `command` / `command_result` | 控制请求、校验结果和生效配置 |
| `alarm` | 仿真生成失败等设备级错误 |

`lidar_raw` 的稳定内部字段包括 `sequence_id`、`frame_id`、`scan_cycle_id`、
`ray_index`、`rays_in_cycle`、角度编码器值、采集/发布偏移、`ranges_m`、主通道、
`channels[]`、退偏比、积分脉冲数和校准状态。这是未来厂家适配器的目标数据模型，
不是对厂家字段名的猜测。

## 仍未知

- TCP/UDP/串口/USB 的真实传输方式、端口和会话握手；
- 二进制或文本帧头、长度、版本、校验、压缩和字节序；
- 原始 ADC/光子计数单位、通道排列、累积方式和无效值；
- 启停、自检、扫描配置、固件升级和故障码；
- 相机编码、时间同步、内外参和文件组织；
- 厂家 L1/L2 字段、反演参数、质控规则和 PM 标定方法。

## 实机适配设计

拿到协议后应新增独立边界层：

```text
VendorTransport
  -> VendorFrameDecoder
  -> VendorProtocolAdapter
  -> LidarProfile / DeviceStatusSnapshot
  -> 现有处理线程和 GUI
```

适配器负责厂家单位、时间戳、通道顺序、无效值和错误码转换；现有处理链继续只消费统一
模型。仿真 JSONL 保留为回归测试替身，不改名冒充实机驱动。

## Golden Sample 验收

每种固件版本至少收集：

1. 启动、自检、状态查询、开始、暂停、恢复、停止的双向抓包；
2. 白天/夜间、近场/远场、晴空/云/污染目标的原始数据；
3. 同时刻厂家上位机导出文件和截图；
4. 通道、距离零点、overlap、偏振与能量标定报告；
5. 断线、丢包、非法命令和设备故障样本。

测试应固定断言帧长度、校验、字段、单位、通道样本、时间误差和厂家产品对照误差。
只有这些测试通过，才能逐项把保真矩阵从“假设/未知”升级为“实机确认”。
